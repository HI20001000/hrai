# 系統資料表關係

本文依據 `server/scripts/database/index.js` 與 `server/index.js` 整理目前資料庫結構、關聯與主要 API 行為。

## 1. 資料庫初始化

預設資料庫名稱為 `hrai`，可由環境變數覆蓋：

- `HRAI_DATABASE`
- `MYSQL_DATABASE`

後端啟動時會：

1. 建立資料庫。
2. 建立帳號表：`users`、`auth_tokens`。
3. 建立 CV、職缺、候選人、項目、人員與 Blacklist 相關表。
4. 對部分既有欄位做相容性補欄，例如 `users.username`、`job_post_applications.remark`、`candidate_blacklist.normalized_phone`。

## 2. 資料表總覽

| 表名 | 主要用途 | 關鍵關聯 |
| --- | --- | --- |
| `users` | 登入帳號、使用者名稱、頭像、密碼 hash/salt。 | 被 `auth_tokens.user_id` 關聯。 |
| `auth_tokens` | 登入 session token hash 與到期時間。 | `user_id -> users.id`。 |
| `candidates` | 候選人主檔：姓名、Email、電話。 | 被 CV、投遞、匹配關聯。 |
| `candidate_cvs` | 候選人 CV 檔案版本與本機 storage key。 | `candidate_id -> candidates.id`。 |
| `candidate_cv_extractions` | CV 原文、AI 分析結果、目標職位。 | `candidate_id -> candidates.id`，`candidate_cv_id -> candidate_cvs.id`。 |
| `job_posts` | 已建立職缺與職位快照。 | 被 `job_post_applications.job_post_id` 關聯。 |
| `candidate_cv_job_matches` | CV 對職位字典的候選匹配結果。 | `candidate_id`、`candidate_cv_id`。 |
| `job_post_applications` | 候選人對具體職缺的投遞與狀態。 | `job_post_id`、`candidate_id`、`candidate_cv_id`。 |
| `personnel` | 內部人員主檔。 | 自關聯主管；被項目與調動關聯。 |
| `projects` | 項目主檔。 | `owner_personnel_id -> personnel.id`。 |
| `project_personnel_assignments` | 項目人員安排。 | `project_id -> projects.id`，`personnel_id -> personnel.id`。 |
| `project_personnel_movements` | 項目人員加入、更新、調動、移出記錄。 | assignment/personnel/project 多方關聯。 |
| `candidate_blacklist` | 候選人 Blacklist。 | 以標準化電話/Email 與候選人命中比對。 |

另外，`finance-job-positions.json` 是職位字典檔案，不是資料表；後端透過 `server/scripts/jobs/dictionary.js` 讀寫。

## 3. 核心關聯圖

### 3.1 帳號

```text
users
  └─< auth_tokens
```

- `auth_tokens.user_id` 外鍵連到 `users.id`。
- 刪除 user 時 token 會 cascade delete。

### 3.2 候選人、CV、職缺、投遞

```text
candidates
  ├─< candidate_cvs
  │    ├─ candidate_cv_extractions
  │    └─< candidate_cv_job_matches
  └─< job_post_applications >─ job_posts
```

重點：

- 一位候選人可以有多個 CV 版本。
- 每個 CV 可有一份 AI 分析結果。
- `job_post_applications` 將候選人、CV 與具體職缺連起來。
- `job_post_applications.candidate_cv_id` 目前有唯一鍵，代表同一份 CV 只會對應一筆職缺投遞。
- 刪除職缺會刪除其投遞，並在程式中清理相關 CV 檔案。
- 刪除候選人會 cascade 刪除 CV、分析、匹配與投遞。

### 3.3 內部人員、項目與調動

```text
personnel
  ├─ personnel.manager_personnel_id -> personnel.id
  ├─< projects.owner_personnel_id
  ├─< project_personnel_assignments >─ projects
  └─< project_personnel_movements
```

重點：

- `personnel` 可自關聯直屬主管。
- `projects.owner_personnel_id` 可指定項目負責人。
- `project_personnel_assignments` 保存目前人員與項目的安排。
- `project_personnel_movements` 保存加入、更新、調動、移出的歷史。

### 3.4 Blacklist

```text
candidate_blacklist
  ├─ normalized_phone
  └─ normalized_email
```

Blacklist 不以外鍵連到 `candidates`，而是透過標準化電話與 Email 進行命中判斷。這樣可以先建黑名單，再於候選人清單中提示命中風險。

## 4. 表結構摘要

### 4.1 `users`

| 欄位 | 說明 |
| --- | --- |
| `id` | 主鍵。 |
| `email` | 登入 email，唯一。 |
| `username` | 顯示名稱。 |
| `avatar_text` | 頭像文字，最多 6 字元。 |
| `avatar_bg_color` | 頭像背景色。 |
| `password_hash` | PBKDF2 hash。 |
| `password_salt` | 密碼 salt。 |
| `created_at` | 建立時間。 |

### 4.2 `auth_tokens`

| 欄位 | 說明 |
| --- | --- |
| `id` | 主鍵。 |
| `user_id` | 對應 `users.id`。 |
| `token_hash` | SHA-256 token hash。 |
| `expires_at` | 到期時間。 |
| `created_at` | 建立時間。 |

### 4.3 `candidates`

| 欄位 | 說明 |
| --- | --- |
| `id` | 主鍵。 |
| `full_name` | 候選人姓名。 |
| `email` | Email。 |
| `phone` | 電話。 |
| `created_at` / `updated_at` | 建立與更新時間。 |

### 4.4 `candidate_cvs`

| 欄位 | 說明 |
| --- | --- |
| `id` | 主鍵。 |
| `candidate_id` | 對應候選人。 |
| `version_no` | 候選人 CV 版本號。 |
| `storage_provider` | 目前預設 `local`。 |
| `storage_key` | 本機檔案 key。 |
| `original_filename` | 原始檔名。 |
| `mime_type` | MIME type。 |
| `file_size` | 檔案大小。 |
| `sha256` | 檔案 hash。 |
| `uploaded_at` | 上傳時間。 |

唯一鍵：`candidate_id + version_no`。

### 4.5 `candidate_cv_extractions`

| 欄位 | 說明 |
| --- | --- |
| `id` | 主鍵。 |
| `candidate_id` | 對應候選人。 |
| `candidate_cv_id` | 對應 CV，唯一。 |
| `target_position` | 從分析結果整理出的目標職位。 |
| `cv_text` | 原始 CV 文字抽取結果。 |
| `extracted_text` | AI 分析 JSON 文字。 |
| `created_at` | 建立時間。 |

### 4.6 `job_posts`

| 欄位 | 說明 |
| --- | --- |
| `id` | 主鍵。 |
| `title` | 職缺標題。 |
| `job_key` | 建立職缺時選用的職位字典 key。 |
| `job_snapshot_json` | 職位字典快照。 |
| `status` | `open`、`draft`、`closed`。 |
| `created_at` / `updated_at` | 建立與更新時間。 |

### 4.7 `candidate_cv_job_matches`

| 欄位 | 說明 |
| --- | --- |
| `candidate_id` | 候選人。 |
| `candidate_cv_id` | CV。 |
| `job_key` / `job_title` | 匹配到的字典職位。 |
| `rank_no` | 排名。 |
| `match_score` | 匹配分數。 |
| `match_level` | 匹配等級。 |
| `reason_summary` | 匹配理由摘要。 |
| `strengths_json` / `gaps_json` | 優勢與差距。 |
| `raw_llm_json` | LLM 原始結果。 |

### 4.8 `job_post_applications`

| 欄位 | 說明 |
| --- | --- |
| `job_post_id` | 對應職缺。 |
| `candidate_id` | 對應候選人。 |
| `candidate_cv_id` | 對應 CV，唯一。 |
| `application_status` | 候選人流程狀態，預設 `screening`。 |
| `first_interview_arrangement` | 一面安排判斷：`can_invite`、`unsuitable` 或空值。 |
| `remark` | 備註。 |
| `matched_score` / `matched_level` | 對該職缺的匹配結果。 |
| `matched_position` | 匹配職位。 |
| `created_at` / `updated_at` | 建立與更新時間。 |

### 4.9 `personnel`

| 欄位 | 說明 |
| --- | --- |
| `full_name` | 內部人員姓名。 |
| `department` / `team` / `title` | 部門、組別、職稱。 |
| `email` / `phone` | 聯絡方式。 |
| `manager_personnel_id` | 直屬主管，指向 `personnel.id`。 |
| `status` | `active` 或 `inactive`。 |
| `remark` | 備註。 |

### 4.10 `projects`

| 欄位 | 說明 |
| --- | --- |
| `project_name` | 項目名稱。 |
| `status` | `planned`、`active`、`paused`、`completed`。 |
| `owner_personnel_id` | 項目負責人。 |
| `start_date` / `end_date` | 起止日期。 |
| `remark` | 備註。 |

### 4.11 `project_personnel_assignments`

| 欄位 | 說明 |
| --- | --- |
| `project_id` | 項目。 |
| `personnel_id` | 人員。 |
| `project_role` | 項目角色。 |
| `start_date` / `end_date` | 入組與離組日期。 |
| `source` | `manual`、`csv`、`candidate` 等來源。 |
| `status` | `active`、`transferred`、`removed`。 |
| `remark` | 備註。 |

唯一鍵：`project_id + personnel_id`。

### 4.12 `project_personnel_movements`

| 欄位 | 說明 |
| --- | --- |
| `assignment_id` | 對應安排，可為 null。 |
| `personnel_id` | 人員。 |
| `from_project_id` / `to_project_id` | 調出與調入項目。 |
| `movement_type` | `added`、`updated`、`transferred`、`removed` 等。 |
| `movement_date` | 調動日期。 |
| `project_role` | 當時角色。 |
| `source` | 來源。 |
| `remark` | 備註。 |

### 4.13 `candidate_blacklist`

| 欄位 | 說明 |
| --- | --- |
| `display_name` | 顯示名稱。 |
| `phone` / `normalized_phone` | 電話與標準化電話。 |
| `email` / `normalized_email` | Email 與標準化 Email。 |
| `reason` | 加入原因。 |
| `status` | `active` 或 `inactive`。 |
| `remark` | 備註。 |

| `POST /api/auth/login` | `users`、`auth_tokens`。 |
| `GET/POST /api/auth/profile` | `users`。 |
| `POST /api/auth/change-password` | `users`。 |

| `GET/POST /api/job-posts` | `job_posts`。 |

| `GET/POST /api/projects` | 項目列表與新增。 |

## 6. 刪除與級聯行為

- `auth_tokens` 隨 `users` 刪除 cascade。
- `candidate_cvs`、`candidate_cv_extractions`、`candidate_cv_job_matches`、`job_post_applications` 多數依候選人或 CV cascade。
- `job_post_applications` 隨 `job_posts` cascade；刪除職缺時程式會先收集 CV storage key 並清理檔案。
- `project_personnel_assignments` 隨 `projects` 或 `personnel` cascade。
- `project_personnel_movements` 對項目採 `ON DELETE SET NULL`，保留歷史記錄。
- `candidate_blacklist` 不與候選人做外鍵綁定。
